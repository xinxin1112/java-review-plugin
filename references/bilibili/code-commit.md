## 三、代码提交规范

### B-001 代码提交规范【推荐】

【推荐】代码提交应该短小而频繁，尽量避免单次提交大量代码。

### B-002 提交说明规范【强制】

【强制】代码提交说明应该描述本次提交的具体内容，并带上适当的前缀：

- `feat`：新功能的说明
- `fix`：修补 bug 的说明
- `docs`：增加文档、注释
- `style`：调整代码格式（不影响任何代码逻辑的变动）
- `refactor`：代码重构（既不是新增功能，也不是修改 bug 的代码变动）
- `test`：增加测试
- `chore`：构建过程或辅助工具的变动

### B-003 提交说明字数【推荐】

【推荐】代码提交说明字数控制在 20 字以内。

### B-004 提交说明标点【推荐】

【推荐】代码提交说明无须以句号结尾。

### B-005 规范化代码提交说明的好处

1. 提供更多的历史信息，方便快速浏览：`git log HEAD --pretty=format:%s`
2. 可以过滤某些 commit（比如功能性提交），便于快速查找信息：`git log HEAD --pretty=format:%s | grep feat`
3. 可以直接从 commit 生成某个 release 的变更日志（参考 conventional-changelog 工具）
4. 可以利用工具自动生成符合规范的 commit message 前缀（参考 commitizen 工具）
5. 对代码审核更加友好，筛选出所有功能性提交（feat 前缀）单独 review 则简单很多

### B-006 表状态变更加锁【强制】

【强制】所有的相关表结构的状态变更，尤其是订单和退款表，Dao 层一定要加锁，建议使用乐观锁提高性能。

### B-107 重复代码强制抽象【强制】

【强制】重复代码超过 4 行（含 4 行）的，必须进行抽象。应将重复逻辑抽取为公共方法、工具类或公共组件，禁止在多处以复制粘贴的方式编写相同或高度相似的代码。

抽象方式（按场景选择）：
- **工具方法：** 无状态的通用逻辑（如参数校验、格式转换）抽取到 `XxxUtil` / `XxxHelper` 类
- **基类/模板方法：** 多个类中流程相同但细节不同的逻辑，使用继承或模板方法模式
- **公共组件/Service：** 跨模块复用的业务逻辑，抽取为独立的公共 Service
- **常量/枚举类：** 重复出现的魔法值、状态码，抽取为常量类或枚举
