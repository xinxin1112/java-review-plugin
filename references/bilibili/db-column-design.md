### 4.3 列设计规范

#### B-023 列设计原则

- 尽量减少存储空间
- 尽量使用数值类型 + `unsigned`

#### B-024 禁用 blob/text 字段【可选】

【尽量避免】尽量避免使用 `BLOB`、`TEXT`、`BINARY` 类型的字段，超大文件建议使用对象存储，在 MySQL 中只保存路径。

**隐患：**

- 会浪费更多的磁盘和内存空间，非必要的大量大字段查询会淘汰掉热数据，导致内存命中率急剧降低，影响数据库性能
- 该表在相同 QPS 下会消耗正常表 N 倍的磁盘 IO 和网络 IO 资源，当 IO 被打满之后，会影响到当前服务器上的所有数据库稳定性

**如果必须使用：**

- 如果所存储字符串个数小于 2000，建议使用 `VARCHAR`
- 必须与主表拆开，使用主键进行关联
- 控制 QPS 在 100 以内
- 一个表只允许一个 `TEXT` 字段
- 存储限制：`TEXT` 2^16 − 1 字节，`MEDIUMTEXT` 2^24 − 1 字节，`LONGTEXT` 2^32 − 1 字节

#### B-025 NOT NULL 约束【必须】

【必须】所有字段都必须使用 `NOT NULL DEFAULT` 属性，避免字段存在 NULL 值。

**默认值规范：**

- 数值类型：`NOT NULL DEFAULT 0`
- 字符类型：`NOT NULL DEFAULT ''`

**为什么必须使用 NOT NULL：**

1. NULL 的列使索引/索引统计/值比较都更加复杂，对 MySQL 来说更难优化
2. NULL 类型 MySQL 内部需要进行特殊处理，增加数据库处理记录的复杂性
3. NULL 值需要更多的存储空间
4. 对 NULL 的处理只能采用 `IS NULL` 或 `IS NOT NULL`，而不能采用 `=`、`IN`、`<`、`<>`、`!=`、`NOT IN` 等操作符
5. 表关联或 WHERE 字段判断时，有 NULL 值会导致返回结果跟实际结果不一致

#### B-026 字段字符集【必须】

【必须】字段不允许设置字符集，继承表字符集即可。

#### B-027 禁止外键【必须】

【必须】禁止使用外键。

#### B-028 数值类型【可选】

【可选】使用 `UNSIGNED` 存储非负整数，已存在字段酌情修改。

#### B-029 禁用 ENUM【强制】

【强制】列的类型不能使用 `ENUM`，强烈建议使用 `TINYINT` 代替。

#### B-030 字符类型规范【强制】

【强制】`CHAR(N)` / `VARCHAR(N)`，N 表示的是字符数不是字节数。在满足够用的前提下，尽可能小的选择 N 的大小，建议 N < 255，N 控制在 4000 以内。

#### B-031 时间类型规范

| 类型 | 说明 | 推荐 |
|------|------|------|
| `timestamp` | 占用 4 字节，有时区概念，最多到 2038 年 | 不推荐 |
| `datetime` | 占用 8 字节，无时区概念，可读性强 | **推荐使用** |
| `int` | 占用 4 字节，无时区概念 | 同 timestamp 范围 |
