# Bilibili 开发规范

> 优先级最高的内置规范。所有代码审查必须首先检查是否符合本规范。

## 三、代码提交规范

### B-001 代码提交规范【推荐】

【推荐】代码提交应该短小而频繁，尽量避免单次提交大量代码。

说明：约束单次提交的范围有利于写出更加针对性的说明，也对代码审核更加友好。

反例：单次提交超过 200+ 行的代码或 20+ 的文件；只在休息的时候（午休、下班）提交代码；很难一句话说清楚这次提交的内容。

### B-002 提交说明规范【强制】

【强制】代码提交说明应该描述本次提交的具体内容，并带上适当的前缀：

- `feat`：新功能的说明
- `fix`：修补 bug 的说明
- `docs`：增加文档、注释
- `style`：调整代码格式（不影响任何代码逻辑的变动）
- `refactor`：代码重构（既不是新增功能，也不是修改 bug 的代码变动）
- `test`：增加测试
- `chore`：构建过程或辅助工具的变动

反例：无意义的说明比如"优化"、"bug fix"、"test"。

### B-003 提交说明字数【推荐】

【推荐】代码提交说明字数控制在 20 字以内。

### B-004 提交说明标点【推荐】

【推荐】代码提交说明无须以句号结尾。

### B-005 规范化代码提交说明的好处

1. 提供更多的历史信息，方便快速浏览：`git log HEAD --pretty=format:%s`
2. 可以过滤某些 commit（比如功能性提交），便于快速查找信息：`git log HEAD --pretty=format:%s | grep feat`
3. 可以直接从 commit 生成某个 release 的变更日志（参考 conventional-changelog 工具）
4. 可以利用工具自动生成符合规范的 commit message 前缀（参考 commitizen 工具）
5. 对代码审核更加友好，筛选出所有功能性提交（feat 前缀）单独 review 则简单很多

### B-006 表状态变更加锁【强制】

【强制】所有的相关表结构的状态变更，尤其是订单和退款表，Dao 层一定要加锁，建议使用乐观锁提高性能。

### B-107 重复代码强制抽象【强制】

【强制】重复代码超过 4 行（含 4 行）的，必须进行抽象。应将重复逻辑抽取为公共方法、工具类或公共组件，禁止在多处以复制粘贴的方式编写相同或高度相似的代码。

说明：重复代码是代码质量劣化的主要根源之一。当同一段逻辑散落在多处时，修改需求或 bug 修复必须同步修改所有副本，遗漏任何一处都会引入不一致和潜在缺陷。

抽象方式（按场景选择）：
- **工具方法：** 无状态的通用逻辑（如参数校验、格式转换）抽取到 `XxxUtil` / `XxxHelper` 类
- **基类/模板方法：** 多个类中流程相同但细节不同的逻辑，使用继承或模板方法模式
- **公共组件/Service：** 跨模块复用的业务逻辑，抽取为独立的公共 Service
- **常量/枚举类：** 重复出现的魔法值、状态码，抽取为常量类或枚举

反例：多个 Service 中出现相同的金额校验逻辑、多个 Mapper 中出现相同的 SQL 片段、多个 Controller 中出现相同的参数组装逻辑。

## 四、数据库规范

> **重要提示：**
> - DB 设计需要过研发内部评审和 review，单个团队发现三次后 DBA 将拒绝处理工单，直至 DB 设计符合规范。
> - 项目提测之前、项目上线之前，强烈建议使用 `explain` 或者 Akso 平台"便捷工具"-"优化建议"检查 SQL 是否用上相关索引。
> - 请提前预留合理的 DB 上线操作时间，有些大表的操作需要数小时才能完成。
> - 以下所有规范针对 InnoDB 存储引擎设计。
> - 对于非标准的数据库（不符合建表规范、不符合授权规范等），无法保障性能稳定性以及实例可用性。

### 4.1 命名规范

#### B-007 数据库命名通用规范

- 命名不要超过 32 个字符
- 采用英文单词/单词缩写/数字，禁用特殊符号，不要在命名中留空格
- 不得采用 `_` 作为名称的起始字母和终止字母
- 命名不得与 MySQL 关键字&保留字冲突（参考 [MySQL 5.7 关键字&保留字](https://dev.mysql.com/doc/refman/5.7/en/keywords.html)）
- 库/表/字段/索引名称一律小写

#### B-008 库名规范

用业务名称命名，做到见名知意，避免使用 `db_`、`kdt_` 开头。单实例单业务，不要混合业务使用数据库。

示例：`tradecenter`、`ump`、`shop`

#### B-009 表名规范

禁止"驼峰"，使用业务命名。

#### B-010 列名规范

避免使用 db 关键字，如 `order`、`type`。

#### B-011 索引命名规范

- 非唯一索引：`ix_字段名称[_字段名称]`
  - 示例：`ix_uid_name`
- 唯一索引：`uk_字段名称[_字段名称]`
  - 示例：`uk_uid_name`

#### B-012 命名风格

C 语言风格（全部小写，`_` 分隔符）。

### 4.2 表设计规范

#### B-014 存储引擎【必须】

【必须】只使用 InnoDB 存储引擎。

#### B-015 物理主键【必须】

【必须】所有表必须有 `INT/BIGINT UNSIGNED NOT NULL AUTO_INCREMENT` 类型的主键，强烈建议该列与业务没有联系，仅作为自增主键 `id` 使用。

**INT / BIGINT 选择标准：**

- 预估数据量在 42 亿条以内：使用 `INT UNSIGNED`
- 预估数据量超过 42 亿条：使用 `BIGINT UNSIGNED`
- 数据增删比较频繁或不确定行数的表：使用 `BIGINT UNSIGNED`
- DBA 后续将提供全局唯一 ID 服务（初始值将超过 INT 上限），建议使用 `BIGINT` 作为物理主键；全局唯一 ID 也是未来双写方案的必要条件

**为什么选择自增 id 作为主键：**

1. 主键自增，数据行写入可以提高插入性能，避免 page 分裂，降低表碎片率，提高磁盘空间利用率
2. 自增型主键（int/bigint）可以降低二级索引的空间，提升二级索引的内存命中率
3. 主键选择较短的数据类型，InnoDB 普通索引都会保存主键的值，较短的数据类型可以有效减少索引磁盘空间，提高索引缓存效率
4. 无主键的表删除，在 row 模式的主从架构下会导致备库夯住

标准格式：

1. `id int unsigned primary key not null auto_increment`
2. `id bigint unsigned primary key not null auto_increment`
3. 主键统一为 `id`，不允许使用 `xxx_id` 等含有业务逻辑的字段作为主键名称

#### B-016 主键约束【必须】

【必须】禁用联合主键，禁止使用字符做主键。

#### B-017 大数据存储【必须】

【必须】不在数据库中存储图片、文件等大数据。

#### B-018 分区表【必须】

【必须】禁止使用分区表。

#### B-019 字符集【必须】

【必须】所有数据库、表除特殊情况外（如表情支持等），都不需要手动指定字符集，统一使用 UTF8MB4。

#### B-020 表和字段注释【必须】

【必须】所有表以及字段必须添加 `COMMENT`，方便自己和他人阅读。一段时间之后可能连自己都不知道这些没有加 COMMENT 的字段是干嘛的。

#### B-021 必加字段【必须】

【必须】所有表必须携带 `ctime`（创建时间）、`mtime`（最后修改时间）这两个字段，便于数据分析以及故障排查。

只需要在建表时建立即可，不需要开发人员再往其中插入时间值（前提是 `INSERT INTO` 语句显式指定字段名称）：

```sql
ctime DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
mtime DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '最后修改时间'
```

**标准建表示例：**

```sql
CREATE TABLE `cloud_bill_analyze_record` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '自增主键ID',
  `cloud_id` varchar(32) NOT NULL DEFAULT '' COMMENT '云平台id',
  `cloud_account` int NOT NULL DEFAULT 0 COMMENT '云平台云账户',
  `cloud_account_name` varchar(64) NOT NULL DEFAULT '' COMMENT '云平台云账户名',
  `collection_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '采集时间',
  `ctime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `mtime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',
  PRIMARY KEY (`id`),
  KEY `ix_mtime` (`mtime`)
) ENGINE=InnoDB COMMENT='云账单分析记录';
```

#### B-022 时间字段索引【可选】

#### B-022 mtime 索引【必须】

【必须】所有表必须将 `mtime` 增加一个普通索引 `ix_mtime(mtime)`，便于数据平台、AI、搜索部门增量获取数据。

【可选】建议同时添加 `ix_ctime(ctime)` 索引。

### 4.3 列设计规范

#### B-023 列设计原则

- 尽量减少存储空间
- 尽量使用数值类型 + `unsigned`

#### B-024 禁用 blob/text 字段【可选】

【尽量避免】尽量避免使用 `BLOB`、`TEXT`、`BINARY` 类型的字段，超大文件建议使用对象存储，在 MySQL 中只保存路径。

**隐患：**

- 会浪费更多的磁盘和内存空间，非必要的大量大字段查询会淘汰掉热数据，导致内存命中率急剧降低，影响数据库性能
- 该表在相同 QPS 下会消耗正常表 N 倍的磁盘 IO 和网络 IO 资源，当 IO 被打满之后，会影响到当前服务器上的所有数据库稳定性

**如果必须使用：**

- 如果所存储字符串个数小于 2000，建议使用 `VARCHAR`
- 必须与主表拆开，使用主键进行关联
- 控制 QPS 在 100 以内
- 一个表只允许一个 `TEXT` 字段
- 存储限制：`TEXT` 2^16 − 1 字节，`MEDIUMTEXT` 2^24 − 1 字节，`LONGTEXT` 2^32 − 1 字节

#### B-025 NOT NULL 约束【必须】

【必须】所有字段都必须使用 `NOT NULL DEFAULT` 属性，避免字段存在 NULL 值。

**默认值规范：**

- 数值类型：`NOT NULL DEFAULT 0`
- 字符类型：`NOT NULL DEFAULT ''`
- 特别注意：`timestamp` 类型不指定默认值时，MariaDB 会默认给 0；多于一个 `timestamp` 字段没有指定默认值，会自动给一个 `timestamp` 默认值为 `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`，其他为 0

**为什么必须使用 NOT NULL：**

1. NULL 的列使索引/索引统计/值比较都更加复杂，对 MySQL 来说更难优化
2. NULL 类型 MySQL 内部需要进行特殊处理，增加数据库处理记录的复杂性；同等条件下，表中有较多空字段时，数据库处理性能会降低很多
3. NULL 值需要更多的存储空间，无论是表还是索引中每行中的 NULL 列都需要额外空间来标识
4. 对 NULL 的处理只能采用 `IS NULL` 或 `IS NOT NULL`，而不能采用 `=`、`IN`、`<`、`<>`、`!=`、`NOT IN` 等操作符。如 `WHERE name != 'bilibili'`，如果存在 name 为 NULL 的记录，查询结果不会包含 name 为 NULL 的记录
5. 表关联或 WHERE 字段判断时，有 NULL 值会导致返回结果跟实际结果不一致

原因：
- NULL 字段很难查询优化
- NULL 字段的复合索引无效
- NULL 字段的索引需要额外空间

#### B-026 字段字符集【必须】

【必须】字段不允许设置字符集，继承表字符集即可。

#### B-027 禁止外键【必须】

【必须】禁止使用外键。

#### B-028 数值类型【可选】

【可选】使用 `UNSIGNED` 存储非负整数，已存在字段酌情修改。注意取值范围：

| 类型 | 存储范围 | 对应时间范围 |
|------|---------|------------|
| `INT` | -2147483648 ~ 2147483647 | 1970/1/1 8:00:00 ~ 2038/1/19 11:14:07 |
| `INT UNSIGNED` | 0 ~ 4294967295 | 1970/1/1 8:00:00 ~ 2106/2/7 14:28:15 |

#### B-029 禁用 ENUM【强制】

【强制】列的类型不能使用 `ENUM`，强烈建议使用 `TINYINT` 代替。新增 ENUM 类型需要 DDL 操作，对于 TINYINT 类型在数据库字段 COMMENT 和程序代码中做好备注信息，避免混淆。

```sql
-- 错误示例：使用 ENUM 类型（数值和字符串插入行为不一致，容易混淆）
CREATE TABLE t (
  id int NOT NULL AUTO_INCREMENT PRIMARY KEY COMMENT '自增ID',
  num enum('0','1','2','3') COMMENT 'enum枚举类型'
);
-- INSERT INTO t(num) VALUES(1)  → 存入 '0'（按索引取值）
-- INSERT INTO t(num) VALUES('1') → 存入 '1'（按字符串匹配）

-- 正确示例：使用 TINYINT 类型
`num` tinyint(4) NOT NULL DEFAULT '0' COMMENT 'TINY枚举类型：0-不通过，1-通过'
```

#### B-030 字符类型规范【强制】

【强制】`CHAR(N)` / `VARCHAR(N)`，N 表示的是字符数不是字节数。在满足够用的前提下，尽可能小的选择 N 的大小，建议 N < 255，N 控制在 4000 以内。

N 尽可能小，因为 MySQL 一个表中所有的 VARCHAR 字段最大长度是 65535 个字节，进行排序和创建临时表一类的内存操作时，会使用 N 的长度申请内存。

#### B-031 时间类型规范

| 类型 | 说明 | 范围 |
|------|------|------|
| `timestamp` | 占用 4 个字节，有时区概念。因为存在时间范围限制，最多表示到 2038 年，故不推荐使用，新表统一选择使用 `datetime` | `'1970-01-01 00:00:00'` ~ `'2038-01-01 00:00:00'` |
| `datetime` | 占用 8 个字节，无时区概念。可读性强，表示时间范围大，**推荐使用** | `'1000-01-01 00:00:00'` ~ `'9999-12-31 23:59:59'` |
| `int` 类型 | 占用 4 个字节，无时区概念。`FROM_UNIXTIME(N)` 和 `timestamp` 范围一致 | 同 `timestamp` |

### 4.4 索引规范

#### B-032 索引目标

利用最小的索引成本找到最需要的行记录。

#### B-033 索引原则

**最左前缀原则：** MySQL 会一直向右匹配直到遇到范围查询（`>`、`<`、`between`、`like`）就停止匹配。例如 `a=1 and b=2 and c>3 and d=4`，如果建立 `(a,b,c,d)` 顺序的索引，`d` 用不到索引；如果建立 `(a,b,d,c)` 的索引则都可以用到，`a,b,d` 的顺序可以任意调整。

**优先使用组合索引：** 避免对每个查询单独创建一个索引，尽量使用组合索引完成索引优化，特别是遇到 `GROUP BY`、`ORDER BY` 条件时，尽可能用组合索引解决。

**避免重复索引：** `idx_abc` 多列索引，相当于创建了 `(a)` 单列索引、`(a,b)` 组合索引以及 `(a,b,c)` 组合索引。

**不在索引列使用函数或表达式：** 如 `max(id) > 10`、`id+1 > 3` 等。

**选择高区分度列作为前缀索引：** 区分度公式为 `count(distinct col)/count(*)`，比例越大扫描的记录数越少。

**上线前根据 SQL 添加索引：** 不仅仅是 `SELECT` 需要添加索引，`UPDATE` 和 `DELETE` 语句也需要添加。

### 4.5 DDL 规范

#### B-034 DDL 变更流程

- 所有变更至少提前一天提交
- 日常变更工作日特定时间之后，重大变更窗口晚上发布
- 所有 DDL 需求请务必由 TL 或者 Owner 审核通过之后提交

#### B-035 新表上线索引【必须】

【必须】所有新上线的表都必须确定索引后才能上线。

### 4.6 SQL 开发规范

#### B-036 禁用存储过程【必须】

【必须】不使用存储过程、触发器、自定义函数。

#### B-037 禁用全文索引【必须】

【必须】不使用全文索引。

#### B-038 多表 Join【必须】

【必须】尽可能不用多表 Join 查询。当确实需要使用多表 Join 时，注意比较字段类型和字符集的一致，避免隐式转换，以及 Join 列有合理索引。

#### B-039 禁用 SELECT *【必须】

【必须】禁用 `select *` 查询所有字段数据，只查询需要的字段数据，尽可能走覆盖索引。

#### B-040 IN 元素个数【可选】

【可选】禁止 `IN` 的元素个数超过 200，防止索引失效导致全表扫描。

#### B-041 禁止前缀模糊查询【必须】

【必须】禁止使用 `%` 前缀模糊查询，如 `where like '%xxx'`。

#### B-042 分页查询【必须】

【必须】遇到分页查询使用延迟关联解决。分页如果有大 offset，可以先取 id，然后用主键 id 关联表会提高效率。

#### B-043 禁止并发 count(*)【必须】

【必须】禁止并发执行 `count(*)`。

#### B-044 禁止负向查询【必须】

【必须】不使用负向查询，如 `not in` / `like`，使用 `in` 反向代替。

#### B-045 禁止 order by rand()【必须】

【必须】禁止使用 `order by rand()`。

#### B-046 OR 改写为 IN【必须】

【必须】SQL 中使用到 `OR` 的改写为用 `IN`（`OR` 的效率没有 `IN` 的效率高）。

#### B-047 查询条件加引号【必须】

【必须】所有查询条件必须加引号，防止类型不对导致的隐式转换。

#### B-048 批量更新/删除【必须】

【必须】不要一次更新大量（大于 30000 条）数据，批量更新/删除。

#### B-049 禁止 force index【必须】

【必须】禁止使用 `force index`。

#### B-050 关联字段类型一致【必须】

【必须】JOIN 查询时，用于 JOIN 的字段定义必须完全相同（避免隐式转换），并且必须建立索引。

#### B-051 禁止 3B 操作【必须】

【必须】禁止 3B 操作：Big Transaction、Big SQL、Big Batch。

#### B-052 指定插入列表【必须】

【必须】必须指定插入列表，即对哪几个列指定插入值，如 `insert into t (id, id2) values(...)`。

#### B-053 避免 replace 语句【建议】

【建议】插入数据避免使用 `replace` 语句，可能会造成唯一键数据重复、增加死锁，且 `replace` 锁复杂。

#### B-054 索引列禁用函数计算【必须】

【必须】避免对索引列进行函数和数值计算，会导致索引失效。

#### B-055 禁止时间类型隐式转换【必须】

【必须】禁止时间类型的隐式转换，如 `where bill_date=DATE('20210228')`（`bill_date` 类型是 `varchar`）。

### 4.7 强烈建议

#### B-056 禁止数据库排序【强烈建议】

【强烈建议】不要在数据库中进行排序，特别是大数据量的排序，可考虑在程序中设计排序。

#### B-057 逻辑删除【强烈建议】

【强烈建议】不要对数据做真正意义的物理删除（`DELETE`），可考虑逻辑删除，即在表中设计一个 `is_deleted` 字段标记该记录是否删除，防止毁灭性事件的发生。

#### B-058 表数据量控制【强烈建议】

【强烈建议】每张表数据量控制在千万级别以下。如果预估超过千万级别，请在设计时考虑归档、日志系统、数据分析平台等方案。

#### B-059 索引选择【强烈建议】

【强烈建议】索引选择时，选择区分度高以及使用频率高的字段，组合索引将其优先放在前面。

### 4.8 尽量避免

#### B-060 避免浮点型类型【尽量避免】

【尽量避免】尽量避免使用浮点型类型，计算机处理整型比浮点型快 N 倍。如果必须使用，请将浮点型扩大 N 倍后转为整型。

#### B-061 避免数据库计算【尽量避免】

【尽量避免】尽量避免在数据库中做计算，减轻数据库压力。

#### B-062 避免 JOIN 查询【尽量避免】

【尽量避免】尽量避免 JOIN 查询，请尽可能使用单表查询，减少查询复杂度，减轻数据库压力。

#### B-063 避免关键字命名【尽量避免】

【尽量避免】尽量避免以 MySQL 关键字&保留字命名（参考 [MySQL 5.7 关键字&保留字](https://dev.mysql.com/doc/refman/5.7/en/keywords.html)）。

#### B-064 避免冗余索引【尽量避免】

【尽量避免】尽量避免出现冗余索引，如索引 `(a)` 和索引 `(a,b)`，此时索引 `(a)` 为冗余索引。

### 4.9 绝对禁止

#### B-065 禁止删除/改名字段【绝对禁止】

【绝对禁止】生产环境中，表一旦设计好，字段只允许增加（`ADD COLUMN`），禁止减少（`DROP COLUMN`、`DROP TABLE`），禁止改名称（`CHANGE/MODIFY COLUMN`）。

#### B-066 禁止 INSERT INTO SELECT【绝对禁止】

【绝对禁止】禁止使用 `INSERT INTO ... SELECT ...` 句式，这种 SQL 会导致锁表。

#### B-067 禁止无序 UPDATE/DELETE LIMIT【绝对禁止】

【绝对禁止】禁止使用 `UPDATE ... LIMIT ...` 和 `DELETE ... LIMIT ...` 操作，因为无法得知究竟更新或删除了哪些数据，请务必添加 `ORDER BY` 进行排序。

```sql
-- 错误示例
UPDATE tb SET col1=value1 LIMIT n;
DELETE FROM tb LIMIT n;

-- 正确示例
UPDATE tb SET col1=value1 ORDER BY id LIMIT n;
DELETE FROM tb ORDER BY id LIMIT n;
```

#### B-068 禁止超过 2 张表 JOIN【绝对禁止】

【绝对禁止】禁止超过 2 张表的 JOIN 查询。

#### B-069 禁止子查询【绝对禁止】

【绝对禁止】禁止使用子查询。

```sql
-- 错误示例
SELECT col1, col2 FROM tb1 WHERE id IN (SELECT id FROM tb2);
```

#### B-070 禁止回退 DDL【绝对禁止】

【绝对禁止】禁止回退表的 DDL 操作。

#### B-071 禁止视图/存储过程/函数/触发器/事件【绝对禁止】

【绝对禁止】禁止在数据库中使用视图、存储过程、函数、触发器、事件。

#### B-072 禁止外键【绝对禁止】

【绝对禁止】禁止使用外键，外键的逻辑应当由程序去控制。

外键会导致表与表之间耦合，`UPDATE` 与 `DELETE` 操作都会涉及相关联的表，十分影响 SQL 的性能，甚至会造成死锁。高并发情况下容易造成数据库性能问题，大数据高并发业务场景数据库使用以性能优先。

#### B-073 禁止 ORDER BY RAND()【绝对禁止】

【绝对禁止】禁止使用 `ORDER BY RAND()` 排序，性能极其低下。

#### B-074 禁止 AFTER/BEFORE 属性【绝对禁止】

【绝对禁止】禁止在添加字段时使用 `AFTER` / `BEFORE` 属性，避免数据偏移。

## 五、语句书写规范

### 5.1 CREATE TABLE 语句

#### B-075 CREATE TABLE 标准写法

```sql
CREATE TABLE `cloud_bill_analyze_record` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '自增主键ID',
  `cloud_id` varchar(32) NOT NULL DEFAULT '' COMMENT '云平台id',
  `cloud_account` int NOT NULL DEFAULT 0 COMMENT '云平台云账户',
  `cloud_account_name` varchar(64) NOT NULL DEFAULT '' COMMENT '云平台云账户名',
  `collection_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '采集时间',
  `ctime` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `mtime` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',
  PRIMARY KEY (`id`),
  KEY `idx_mtime` (`mtime`)
) ENGINE=InnoDB COMMENT='云账单分析记录';
```

### 5.2 ALTER TABLE 语句

#### B-076 ALTER TABLE 规范

> **注意：** 为了兼容下游 Hudi、DTS 等系统，打通上下游数据 Binlog 同步链条，MySQL/TiDB 修改表结构语句规范从 2022/02/25 开始进行了如下调整：
> - 禁止变更字段类型（`int` 升级为 `bigint` 以及修改原类型的长度不受限制）
> - 禁止修改列名称、列顺序
> - 禁止 `DROP` 列

**添加字段：**

```sql
-- 正确示例
ALTER TABLE bilibili ADD COLUMN c8 int(11) NOT NULL DEFAULT 0 COMMENT '添加字段测试';
```

添加字段时禁止使用 `AFTER` / `BEFORE` 属性，避免数据偏移。

**变更字段（MODIFY 只修改字段定义）：**

```sql
-- 正确示例
ALTER TABLE bilibili MODIFY COLUMN c8 varchar(16) NOT NULL DEFAULT 0 COMMENT 'MODIFY修改字段定义';
```

**添加主键：**

```sql
-- 正确示例
ALTER TABLE t5 ADD PRIMARY KEY(id), MODIFY id bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '自增主键ID';
```

**修改字符集：**

```sql
-- 正确示例
ALTER TABLE tablename CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_bin;
```

**注意：** 同一张表的 DDL 建议整合到一个 SQL 里面，否则表会重建 N 次。

### 5.3 CREATE/DROP INDEX 语句

#### B-077 索引操作规范

> **注意：** 索引升序 `ASC`，降序 `DESC`。MariaDB 暂时不支持降序索引 `DESC`，之所以能创建成功是因为其将 `DESC` 设置为关键字，实际上仍然是 `ASC` 索引。

**添加普通索引：**

```sql
-- 正确示例（联合索引时，筛选度高的列靠左）
ALTER TABLE tb1 ADD INDEX idx_c3(c3);
```

**添加唯一索引：**

```sql
-- 正确示例
ALTER TABLE tb1 ADD UNIQUE INDEX uk_c2(c2);
```

**删除普通索引：**

```sql
-- 正确示例
ALTER TABLE tb1 DROP INDEX idx_c3;
```

**删除唯一索引：**

```sql
-- 正确示例
ALTER TABLE tb1 DROP INDEX uk_c2;
```

> **注意：** 无论测试环境与生产环境，表结构变更以及访问账号的权限均需要提交工单。

### 5.4 SELECT 语句

#### B-078 SELECT 规范

【禁止】禁止使用 `SELECT * FROM` 语句，`SELECT` 只获取需要的字段，既防止了新增字段对程序应用逻辑的影响，又减少了对程序和数据库的性能影响。

```sql
-- 错误示例
SELECT * FROM tb WHERE col1=value1;
-- 正确示例
SELECT col1, col2 FROM tb WHERE col1=value1;
```

【必须】合理使用数据类型，避免出现隐式转换，隐式转换无法使用索引且效率低。

```sql
-- 错误示例（age 为 varchar 类型，传入 int 导致隐式转换）
SELECT name FROM tb WHERE age=1;
```

【禁止】不建议使用 `%` 前缀模糊查询，导致查询无法使用索引。

```sql
-- 错误示例
SELECT id FROM tb WHERE name LIKE '%bilibili';
```

【强烈建议】对于 `LIMIT` 操作，强烈建议先 `ORDER BY` 再 `LIMIT`，即 `ORDER BY c1 LIMIT n`。

### 5.5 INSERT 语句

#### B-079 INSERT 规范

【必须】`INSERT INTO` 语句需要显式指明字段名称。

```sql
-- 正确示例
INSERT INTO tb(col1, col2) VALUES(value1, value2);
```

【必须】对于多次单条 `INSERT INTO` 语句，务必使用小批量 `INSERT INTO` 语句（一般控制在 200 条以内一批），提高效率。

```sql
-- 错误示例：多次单条 INSERT
INSERT INTO tb(col1, col2) VALUES(value1, value2);
INSERT INTO tb(col1, col2) VALUES(value3, value4);
INSERT INTO tb(col1, col2) VALUES(value5, value6);

-- 正确示例：批量 INSERT
INSERT INTO tb(col1, col2) VALUES(value1, value2),(value3, value4),(value5, value6);
```

### 5.6 UPDATE 语句

#### B-080 UPDATE 规范

```sql
-- 正确示例
UPDATE tb SET col1=value1, col2=value2, col3=value3 WHERE col0=value0 AND col5=value5;
```

> **注意：** `SET` 后接的并列字段分隔符为逗号（`,`），而不是 `AND`。使用 `AND` 也能将 UPDATE 语句执行成功，但意义完全不一样（`col1=value1 AND col2=value2` 会被解析为布尔表达式赋值给 `col1`）。

- 【强烈建议】UPDATE 语句后携带 `WHERE` 条件，防止灾难性事件的发生
- 【强烈建议】如果需要使用 UPDATE 修改大量数据时，建议以主键为条件更新，每次更新 1000-3000 条，中间 sleep 1-3s，提交事务，循环更新。防止造成大量锁定以及主从复制延迟
- 【禁止】禁止使用 `UPDATE ... LIMIT ...` 语法（参见 B-067）

### 5.7 DELETE 语句

#### B-081 DELETE 规范

```sql
-- 正确示例
DELETE FROM tb WHERE col0=value0 AND col1=value1;
```

- 【强烈建议】DELETE 语句后携带 `WHERE` 条件，防止灾难性事件的发生
- 【强烈建议】如果需要使用 DELETE 删除大量数据时，建议以主键为条件删除，每次删除 1000-3000 条，中间 sleep 1-3s，提交事务，循环删除。防止造成大量锁定以及主从复制延迟
- 【禁止】禁止使用 `DELETE ... LIMIT ...` 语法（参见 B-067）

### 5.8 其他书写规范

#### B-082 禁止在字段上使用函数

【禁止】禁止在字段上使用函数，会导致索引失效。

```sql
-- 错误示例
SELECT col1, col2 FROM tb WHERE unix_timestamp(col1)=value1;
-- 正确示例
SELECT col1, col2 FROM tb WHERE col1=unix_timestamp(value1);
```

#### B-083 字段放在操作符左边

【强烈建议】字段放在操作符左边。

```sql
-- 错误示例
SELECT col1, col2 FROM tb WHERE value1=col1;
-- 正确示例
SELECT col1, col2 FROM tb WHERE col1=value1;
```

#### B-084 禁止隐式类型转换

【禁止】禁止将字符类型传入到整型类型字段中，也禁止整型类型传入到字符类型字段中。字符类型字段存在隐式转换问题，会导致索引失效。

```sql
-- 错误示例
-- var_col 字段为 VARCHAR 类型
SELECT col1, col2 FROM tb WHERE var_col=123;
-- int_col 字段为 INT 类型
SELECT col1, col2 FROM tb WHERE int_col='123';

-- 正确示例
-- var_col 字段为 VARCHAR 类型
SELECT col1, col2 FROM tb WHERE var_col='123';
-- int_col 字段为 INT 类型
SELECT col1, col2 FROM tb WHERE int_col=123;
```

## 六、程序操作数据库设置规范

### 6.1 必须遵守

#### B-085 长连接自动重连【必须】

【必须】如果应用使用的是长连接，应用必须具有自动重连的机制，但请避免每执行一个 SQL 去检查一次 DB 可用性。

#### B-086 长连接 TIMEOUT 检查【必须】

【必须】如果应用使用的是长连接，应用应该具有连接的 TIMEOUT 检查机制，及时回收长时间没有使用的连接，TIMEOUT 时间一般建议为 2 小时。

#### B-087 连接字符集【必须】

【必须】程序访问数据库连接的字符集请设置为 `utf8mb4`。

### 6.2 绝对禁止

#### B-088 禁止程序中 DDL【绝对禁止】

【绝对禁止】程序中禁止一切 DDL 操作。

## 七、行为规范

#### B-089 禁止手工访问线上数据库【禁止】

【禁止】禁止使用应用程序配置文件内的帐号手工访问线上数据库。大部分配置文件内的数据库配置的是主库，无法预知一条 SQL 会不会导致 MySQL 崩溃。

#### B-090 大型活动提前沟通【必须】

【必须】大型活动（如拜年祭）或突发性大量操作数据库（如发送私信）等操作时，应提前与 DBA 当面沟通，进行流量评估，避免数据库出现瓶颈。

#### B-091 批量清洗数据【必须】

【必须】批量清洗数据，需要开发和 DBA 共同进行审查，应避开业务高峰期时段执行，并在执行过程中观察服务状态。

#### B-092 禁止主库执行统计查询【禁止】

【禁止】禁止在主库上执行后台管理和统计类的功能查询，这种复杂类的 SQL 会造成 CPU 升高，进而影响业务。

## 八、分库分表命名规则

#### B-093 分库分表命名规范

**自增数字分表（库）：** 表（库）名使用自动补齐规则：

- 分 10 张表：`bilibili_0` ~ `bilibili_9`
- 分 100 张表：`bilibili_00` ~ `bilibili_99`
- 分 1000 张表：`bilibili_000` ~ `bilibili_999`

**按年分表（库）：** 后缀为对应年份：

- `bilibili_2017` ~ `bilibili_2020`

**按月分表（库）：** 后缀为对应年月：

- `bilibili_201701` ~ `bilibili_202012`

**按天分表（库）：** 后缀为对应年月日：

- `bilibili_20170101` ~ `bilibili_20201201`

## 九、常用字段数据类型范围

### 9.1 数值类型

| 数值类型 | 取值范围 |
|---------|---------|
| `TINYINT(4)` | -128 ~ 127 |
| `TINYINT(4) UNSIGNED` | 0 ~ 255 |
| `SMALLINT(6)` | -32768 ~ 32767 |
| `SMALLINT(6) UNSIGNED` | 0 ~ 65535 |
| `MEDIUMINT(8)` | -8388608 ~ 8388607 |
| `MEDIUMINT(8) UNSIGNED` | 0 ~ 16777215 |
| `INT(11)` | -2147483648 ~ 2147483647 |
| `INT(11) UNSIGNED` | 0 ~ 4294967295 |
| `BIGINT(20)` | -9223372036854775808 ~ 9223372036854775807 |
| `BIGINT(20) UNSIGNED` | 0 ~ 18446744073709551615 |

### 9.2 字符类型

`VARCHAR(N)`：在 MySQL 中，N 代表 N 个字符（不是字节），不管中文字符还是英文字符，`VARCHAR(N)` 能存储最大为 N 个字符。

### 9.3 时间类型

| 类型 | 取值范围 | 备注 |
|------|---------|------|
| `TIMESTAMP` | `1970-01-01 00:00:01 UTC` ~ `2038-01-19 03:14:07 UTC` | 会调用系统时区转化，大规模并发时会影响性能 |
| `DATETIME` | `1000-01-01 00:00:00` ~ `9999-12-31 23:59:59` | 推荐使用 |

## 十、CR 规范

### 10.1 时间点

- **迭代版本：** 提测前
- **独立项目：** 进入开发周后，每周一次

### 10.2 关注点

需与各团队同步细节，一起完善。评审过程中，可参考的事项列表：

#### B-094 设计审查

代码是否经过精心设计并适合您的系统？包含业务严谨性，这部分组内相关 Owner 重点跟进。

#### B-095 功能审查

代码的行为是否与需求目标一致？

#### B-096 复杂度审查

代码能更简单吗？将来其他开发人员能轻松理解并使用此代码吗？

其中一种复杂性就是过度工程（over-engineering），如开发人员使代码过度通用，超过它原本所需的，或者添加系统当前不需要的功能。审查者应特别警惕过度工程。未来的问题应该在它实际到达后解决，且届时才能更清晰地看到其真实样貌及在现实环境里的需求。鼓励开发人员解决他们现在需要解决的问题，而不是开发人员推测可能需要在未来解决的问题。

#### B-097 测试审查

代码是否具有正确且设计良好的单测、自动化测试？

#### B-098 命名审查

开发人员是否为变量、类、方法等选择了明确的名称？

#### B-099 注释审查

注释是否清晰有用？

通常，注释应该用于解释代码的用处，而不是解释它们在干什么。如果代码不够清晰，无法自解释，那就应该简化代码。当然也有一些例外（例如正则表达式和复杂的算法，如果能够解释它们在做什么，会让阅读代码的人受益匪浅），但大多数注释都应该指出代码中不可能包含的信息，比如这些代码背后的缘由。

#### B-100 风格审查

代码是否遵守了风格指南？每个项目组架构风格存在一些差异，自行完善。Java 基础编程需遵循【代码规范】风格。

#### B-101 文档审查

开发人员是否同时更新了相关文档？变更了相关接口、业务逻辑，或者删除、弃用了代码，需考虑相关文档是否更新。

#### B-102 日志、SQL、MQ 审查

DDL、MQ 变更或新增是否合理。

#### B-103 可修改性检查

代码涉及到的常量是否易于修改（如使用配置、定义为类常量、使用专门的常量类等）。

#### B-104 健壮性检查

- **异常处理：** 采取措施避免运行时错误（如空指针、数组边界溢出、被零除、值越界、堆栈溢出等）
- **性能问题：** 是否存在代码中断的特定情况（例如峰值加载次数）？查询是否拉取了比所需更多的数据？向数据库中增加新索引是否有助于新查询？
- **依赖关系：** 是否引入了外部库或者 API？是否有其它用不同依赖或者没有依赖的更简单/更快速/更好的方法来实现？

#### B-105 安全性检查

进行身份验证、授权、输入数据验证，避免诸如 SQL 注入和跨站脚本（XSS）等安全威胁，加密敏感数据（密码、信用卡信息等）。

#### B-106 可重用性检查

- **DRY 原则（Do not Repeat Yourself）：** 同一代码不应该重复两次以上
- **重复代码超过 4 行必须抽象**（参见 B-107）
- 考虑可重用的服务、功能和组件
- 考虑通用函数和类

### 10.3 总结

在进行代码审查时，您应该确保：

- 代码设计良好
- 其中任何并行编程都是安全的
- 代码复杂性不要超过应有的程度
- 开发人员没有实现他们将来可能需要，但不知道他们现在是否需要的东西
- 代码有适当的单元测试，精心设计的测试用例
- 清晰的命名方式
- 清晰有用的代码注释，要解释为什么而不是做什么
- 代码符合风格指南

### 10.4 执行

#### 10.4.1 事前-准备

1. 至少提前一天拉取 CR 会议，包含 CR 内容、设计方案等材料
2. 选择 CR 活动的参与者，除项目组相关人员，要邀请业务架构 & 业务稳定性小组同学
3. 每个版本可以按业务域拆开评审，不建议一次评审时间太久

#### 10.4.2 事中-实施

1. 需求背景、设计文档、方案、接口文档讲解
2. 根据上述文档以及代码改动点列举出所有受影响的业务场景
3. 根据第二点 review 所有单测 case
4. 运行单测用例
5. 开始 CR：
   - 指定 CR 记录人员，准确记录
   - 对于 CR 过程发现的问题，必须清晰准确的记录，详见问题记录单

**讲解与提问：** CR 过程中，要采用代码作者讲解和审查者提问方式。审查者不能只在发现问题时提问，同时也要根据本次审查的内容要求代码作者对某个特定问题的讲解。

**注意气氛：** 实施审查时，要营造一个讨论问题、解决问题的氛围，不能把审查会搞成批判会，这样会影响相关人员的积极性。

#### 10.4.3 事后-跟踪

1. 针对已发现的问题，确定解决人以及解决时间点，二次 CR 时间。问题记录清单发送到相应群组。
2. 针对问题点以及 good case，更新代码规范、最佳实践相关知识库。技术 Owner 负责完善相关知识库（版本里的技术 Owner、独立项目的技术 Owner）。
3. 最终改动需在上线前三天完成，给测试预留回归时间。

## 十一、注意事项

全面快速的代码评审需要技术技能。但除此之外，代码评审也是一种沟通、教学和学习的形式。无论是作为代码的作者还是评审者，在沟通中考虑周到可以使代码评审对每个人都更有价值。

### 11.1 良好的氛围

**礼貌：** 不要急于下结论，要多问问题，即使乍一看似乎完全是错的。要确保你的评论是针对代码而不是针对开发人员。例如：

- 不好的说法："为什么你要在这个地方使用线程，这样做显然不会获得任何好处"
- 好的说法："在这里使用并发模型会增加系统复杂性，它带来的实际意义或者好处是什么呢？"

**解释你的理由：** 有助于开发人员理解你为什么要给出这些建议。如果你能够为你的意图、所遵循的最佳实践或你的建议将如何改进代码质量给出更多的解释会更好。

**给出明确的方向：** 指出问题，并让开发人员决定如何在两者之间做出权衡。这样有助于开发人员学到东西，并让代码评审变得更容易，还可以产出更好的解决方案，因为开发人员比评审人员更了解代码。不过，有时候直接给出指令、建议或代码会更有用。代码评审的主要目的是获得尽可能好的 CL，第二个目的是提高开发人员的技能。

**鼓励简化：** 鼓励开发人员简化代码，或者添加代码注释，而不只是让他们解释代码的复杂性。

**给予表扬：** 也许你从评审中学到了一些东西，或者对一些细节让你印象深刻，可以说出来，让他知道，并记为 good case。对新手来说，在代码评审中给予表扬尤其重要。评审者应该同理心看待，新加入者可能还不知道所有的编程风格，并且可能对代码的某些部分还不熟。良好的代码评审氛围会让新加入者的前几次评审成为一次很好的体验。

### 11.2 保持积极正面的态度

无论是代码作者还是评审者，都需要一种积极向上的正面态度。作者需要能够虚心接受别人的建议，因为别人的建议是为了让你做得更好；评审者也需要以一种积极的正面的态度向作者提意见，因为那是和你在一个战壕里的战友。记住，你不是一段代码，你是一个人！

### 11.3 学会享受 Code Review

CR 的过程实际就是一个互相进步的过程，一次免费的学习机会——仅仅通过评审别人的代码，你就能更熟练地阅读和理解新代码。杰出的工程师最重要的特质之一就是深入研究和剖析一段完全陌生代码的能力。随着时间的推移，你将学会如何发现常见的实践、小技巧、语法糖、架构抽象，以及如何欣赏用来解决同一问题的不同思维模式。

### 11.4 Code Review 不应该成为保证代码风格和编码标准的手段

编码风格和代码规范都属于死的东西，每个程序员在把自己的代码提交团队 Review 的时候，代码就应该是符合规范的，这是默认值，属于每个人自己的事情，不应该交由团队来完成，否则只会浪费大家本来就不够的时间。

### B-013 个人信息字段命名规范

多列场景统一用 `{数据分类}_{数据作用}` 命名。

| 类型 | 字段 | 多列命名规则 |
|------|------|------------|
| 地址 | `address` | `{数据分类}_address`（如 `buyer_address`、`seller_address`） |
| 真实姓名 | `true_name` | `{数据分类}_true_name`（如 `buyer_true_name`） |
| 手机号 | `mobile` | `{数据分类}_mobile`（如 `buyer_mobile`） |
| 身份证号 | `card_number` | `{数据分类}_card_number`（不用 `id_card`，因 id 在多字段有应用） |
| 身份证照片 | `card_photo` | 正面 `front_card_photo`、反面 `back_card_photo`、手持 `hold_card_photo` |
| 邮箱 | `mail` | `{数据分类}_mail`（如 `buyer_mail`） |
| 支付账号 | `pay_account` | `{数据分类}_pay_account`（如 `buyer_pay_account`） |
| 银行卡号 | `bank_card` | `{数据分类}_bank_card`（如 `buyer_bank_card`） |
| 支付宝唯一 ID | `pay_account_id` | `pay_account_id` |
| 护照信息 | `passport_number` | 号码 `passport_number`、照片 `passport_photo`、手持 `hold_passport_photo` |
| 密码/口令 | `password` | `{数据分类}_password`（如 `account_password`、`pay_password`） |
